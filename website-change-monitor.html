Website Change Monitor and Alert

On a schedule, Bright Data Web Unlocker fetches a target URL (even JavaScript-rendered pages that most monitoring tools miss). The content is compared against the previous snapshot, AI summarizes what changed, and your team gets alerted on Slack or email. Works with Airtable, Notion, or any database for snapshot storage.

Zapier template coming soon. Step-by-step guide in progress.
What this scenario does: On a schedule, the scenario reads all active URLs from your storage (Airtable by default), uses Bright Data Web Unlocker to fetch the current HTML of each one (handles JS-rendered pages, CAPTCHAs, anti-bot), compares it against the previous snapshot, and uses Claude to detect real visible changes (filtering out structural noise like timestamps, build IDs, CSS). When something genuinely changed, it updates the snapshot and fires a Slack alert with a 2-3 sentence summary.

5 modules: Airtable (Search Records) → Bright Data (Web Unlocker) → Claude (Compare HTML) → Airtable (Update Record) → Slack (Send Alert)

Get Make template
Storage and alerts are swappable: the template uses Airtable for storage and Slack for alerts by default, but both are easy to replace. Storage can be Notion, Google Sheets, Postgres, MySQL, MongoDB, Supabase, or any database. Alerts can go to Discord, Teams, Telegram, Gmail, SMS, or any messaging app.
1
Activate Web Unlocker on Bright Data
This is the killer feature for this template - Web Unlocker handles JavaScript-rendered pages, CAPTCHAs, and bot detection that other monitoring tools (Visualping, Distill, Wachete) miss.

  • Log in at brightdata.com
  • Sidebar → Proxies & Scraping Infrastructure
  • Click Add → select Web Unlocker
  • Click Start with Web Unlocker or Create zone
  • Name it web_unlocker_agent (or any name) and save
Full docs: Web Unlocker introduction.
2
Set up your storage - any database works
The storage holds the URLs to monitor and the previous HTML snapshot of each. Pick whatever your team uses:
  • Airtable - template default, easiest UI to manage URLs
  • Notion - database with the same fields
  • Google Sheets - same fields as columns
  • Postgres / MySQL / MongoDB / Supabase - any database via Make's native modules
Whatever you pick, the storage needs these fields:
  • URL (text) - the page to monitor
  • Label (text) - human-friendly name (e.g. "Stripe Pricing Page")
  • Last HTML (long text) - previous snapshot (filled by the scenario)
  • Last Checked At (date) - timestamp (filled by the scenario)
  • Last Change Summary (long text) - last AI summary (filled by the scenario)
  • Active (checkbox) - toggle to enable/disable monitoring per URL
The template ships with Airtable. If you swap it, replace the Airtable modules in steps 3 and 6 with the equivalent Read/Update modules from your storage.
3
Module 1 - Airtable Search Records (read all active URLs)
The first module is Airtable - Search Records. It reads all active URLs from your URLs table.

Pre-configured filter: {Active} = TRUE() with Limit = 100. Connect your Airtable account, point to your base + table.

Schedule: set the run frequency in Make's scenario settings (recommended: every 1-6 hours depending on your needs).
4
Module 2 - Bright Data Web Unlocker (fetch current HTML)
For each URL from the trigger, this module fetches the current page content via Web Unlocker. Pre-configured:
  • Zone → web_unlocker_agent
  • URL → {{1.URL}} from Airtable
  • Format → raw
  • Method → GET
Connect your Bright Data account. Web Unlocker handles JS rendering, CAPTCHAs, bot detection automatically.
5
Module 3 - Claude (smart change detection)
This is what makes the template smart. Most monitoring tools just compare hashes - they alert you for every meaningless change (timestamps, build IDs, dynamic CSS).

This module uses Claude Sonnet 4.6 to compare the previous HTML vs the current HTML and only flag real visible changes: pricing changes, new/removed features, copy updates, button labels, new sections.

Filter on this module: only runs if {{2.result}} != {{1.Last HTML}} (avoids running Claude when nothing changed at all).

The prompt instructs Claude to return:
  • FIRST_RUN if there's no previous snapshot
  • NO_REAL_CHANGE if only structural noise changed
  • A 2-3 sentence summary if there's a real change
These special tokens are used by the alert filter in step 7.
6
Module 4 - Update the storage (write new snapshot)
Updates the Airtable record with:
  • Last HTML → the new HTML (truncated to 99000 chars to fit Airtable's limit)
  • Last Change Summary → Claude's response
  • Last Checked At → {{now}}
If you swapped the storage (Notion / Sheets / DB): replace this module with the equivalent Update / Upsert action and map the same three fields.
7
Module 5 - Send the alert (any channel works)
The alert can go anywhere your team listens. Pick whatever fits:
  • Slack - template default, mrkdwn formatting
  • Discord / Microsoft Teams / Telegram - Send Message
  • Gmail / Outlook / SMTP - email the alert
  • Twilio - SMS or WhatsApp
  • PagerDuty - for critical pages
  • Linear / Jira - turn changes into tickets
The template ships with Slack using this format:
🔔 Website change detected: *{{Label}}* 🔗 {{URL}} 📝 What changed: {{Claude summary}} ⏰ Detected: {{now}}
The smart filter on this module is the key: it only fires if Claude's summary is NOT FIRST_RUN and NOT NO_REAL_CHANGE. So you only get alerts for actual visible changes - no noise from timestamps, build IDs, or dynamic CSS.
8
Test and activate
Add a few URLs to your Airtable with Active = true. Click Run once in Make. The first run for each URL will save the initial snapshot (Claude returns FIRST_RUN, no Slack alert). On subsequent runs, real changes trigger Slack alerts.

Set the schedule in Make (every hour, 6 hours, daily) and toggle the scenario ON.

Troubleshooting

ProblemSolution
Web Unlocker 403 / timeoutMake sure your Web Unlocker zone is created and active. For sites with heavy JS, use a zone with rendering enabled.
Too many false alertsThe AI filter should remove most noise. If you still get noise, tighten the prompt: "Ignore: navigation menus, footer links, social icons."
Missing real changesSwitch Claude model to Opus 4.7 for sharper detection. Or pre-strip HTML to extract just the main content (use a Text Parser before Claude).
Airtable HTML field too longThe template already truncates to 99000 chars. For larger pages, use a long-text field in Notion or store as a file in S3 / Google Drive instead.
Want to monitor specific sections onlyAdd a Text Parser module after Web Unlocker to extract the relevant section by CSS selector (e.g. div.pricing-table) before sending to Claude.
What this workflow does: On a schedule, n8n reads all active URLs from your storage (Airtable by default), loops through each one, uses Bright Data Web Unlocker to fetch the current HTML (handles JS, CAPTCHAs, anti-bot, geo-blocks), truncates it to fit the LLM context, and uses Claude Sonnet 4.6 to detect real visible changes. The new snapshot is saved back to your CRM. If a real change is detected (not FIRST_RUN or NO_CHANGE), Slack gets alerted with a 2-3 sentence summary.

Node 3 - Loop Over URLs (Split In Batches)
The Loop Over URLs node processes one URL at a time. This keeps the workflow clean and resilient - if one URL fails (timeout, 404), the others still get processed.

No setup needed. The node just iterates through the records returned by the previous node.
6
Node 4 - Scrape Page with Bright Data
The Bright Data Web Unlocker node fetches the current page content. Pre-configured:

  • Country → us (change if you need a different geo)
Connect your Bright Data credentials. Web Unlocker handles JS rendering, CAPTCHAs, and bot detection automatically.

Node 5 - Truncate HTML
A small Code node keeps only the first 90,000 characters of the HTML to fit Claude's context window.

No setup needed unless you want to adjust the limit (you can increase it for larger pages or models with bigger context windows).
